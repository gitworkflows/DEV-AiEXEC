:::warning
Never expose Aiexec ports directly to the internet without proper security measures.
Set `AIEXEC_AUTO_LOGIN=False`, use a non-default `AIEXEC_SECRET_KEY`, and deploy your Aiexec server behind a reverse proxy with authentication enabled.
For more information, see [Start a Aiexec server with authentication enabled](#start-a-aiexec-server-with-authentication-enabled).
:::

Authentication credentials help prevent unauthorized access to your Aiexec server, flows, and services connected through components.

There are three types of credentials that you use in Aiexec:

* [Aiexec API keys](#aiexec-api-keys): For authentication with the Aiexec API and authorizing server-side Aiexec actions like running flows and uploading files.
* [Component API keys](#component-api-keys): For authentication between Aiexec and a service connected through a component, such as a model provider or third-party API.
* [Authentication environment variables](#authentication-environment-variables): These environment variables configure how Aiexec handles user authentication and authorization.

## Aiexec API keys {#aiexec-api-keys}

You can use Aiexec API keys to interact with Aiexec programmatically.

In Aiexec version 1.5 and later, most API endpoints require a Aiexec API key, even when `AIEXEC_AUTO_LOGIN` is `True`.
For more information, see [`AIEXEC_AUTO_LOGIN`](#aiexec-auto-login).

### Aiexec API key permissions

A Aiexec API key adopts the privileges of the user who created it.
This means that API keys you create have the same permissions and access that you do, including access to your flows, components, and Aiexec database.
A Aiexec API key cannot be used to access resources outside of your own Aiexec server.

In single-user environments, you are always a superuser, and your Aiexec API keys always have superuser privileges.

In multi-user environments, users who aren't superusers cannot use their API keys to access other users' resources.
You must [start your Aiexec server with authentication enabled](#start-a-aiexec-server-with-authentication-enabled) to allow user management and creation of non-superuser accounts.

### Create a Aiexec API key

You can generate a Aiexec API key in your Aiexec **Settings** or with the Aiexec CLI.

The CLI option is required if your Aiexec server is running in `--backend-only` mode.

<Tabs>
<TabItem value="settings" label="Aiexec Settings" default>

1. In the Aiexec header, click your profile icon, and then select **Settings**.
2. Click **Aiexec API Keys**, and then click **Add New**.
3. Name your key, and then click **Create API Key**.
4. Copy the API key and store it securely.

</TabItem>
<TabItem value="Aiexec CLI" label="Aiexec CLI">

If you're serving your flow with `--backend-only=true`, you can't create API keys in your Aiexec **Settings** because the frontend isn't running.
In this case, you must create API keys with the Aiexec CLI.

1. Recommended: [Start your Aiexec server with authentication enabled](#start-a-aiexec-server-with-authentication-enabled).

    This configuration is recommended for security reasons to prevent unauthorized API key and superuser creation, especially in production environments.

    However, if authentication isn't enabled (`AIEXEC_AUTO_LOGIN=True`), all users are effectively superusers, and they can create API keys with the Aiexec CLI.

2. Create an API key with [`aiexec api-key`](/configuration-cli#aiexec-api-key):

    ```shell
    uv run aiexec api-key
    ```

    All API keys created with the Aiexec CLI have superuser privileges because the command requires superuser authentication, and Aiexec API keys adopt the privileges of the user who created them.

</TabItem>
</Tabs>

### Use a Aiexec API key

To authenticate Aiexec API requests, pass your Aiexec API key an `x-api-key` header or query parameter.

<Tabs>
<TabItem value="header" label="HTTP header" default>

```shell
curl -X POST \
  "http://$AIEXEC_SERVER_ADDRESS/api/v1/run/$FLOW_ID?stream=false" \
  -H "Content-Type: application/json" \
  -H "x-api-key: $AIEXEC_API_KEY" \
  -d '{"inputs": {"text":""}, "tweaks": {}}'
```

</TabItem>
<TabItem value="parameter" label="Query parameter">

```shell
curl -X POST \
  "http://$AIEXEC_SERVER_ADDRESS/api/v1/run/$FLOW_ID?x-api-key=$AIEXEC_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"inputs": {"text":""}, "tweaks": {}}'
```

</TabItem>
</Tabs>

For more information about forming Aiexec API requests, see [Get started with the Aiexec API](/api-reference-api-examples) and [Trigger flows with the Aiexec API](/concepts-publish).

### Revoke an API key

To revoke and delete an API key, do the following:

1. In the Aiexec header, click your profile icon, and then select **Settings**.
2. Click **Aiexec API Keys**.
3. Select the keys you want to delete, and then click <Icon name="Trash2" aria-hidden="true"/> **Delete**.

This action immediately invalidates the key and prevents it from being used again.

## Component API keys

Component API keys authorize access to external services that are called by components in your flows, such as model providers, databases, or third-party APIs.
These aren't Aiexec API keys or general application credentials.

In Aiexec, you can store component API keys in global variables in your **Settings** or import them from your Aiexec `.env` file.
When creating global variables, use the **Credential** type for secure handling of sensitive information.
For more information, see [Global variables](/configuration-global-variables).

You create and manage component API keys within the service provider's platform.
Aiexec only stores the encrypted key value or a secure reference to a key stored elsewhere; it doesn't manage the actual credentials at the source.
This means that deleting a global variable from Aiexec doesn't delete or invalidate the actual API key in the service provider's system.
You must delete or rotate component API keys directly using the service provider's interface or API.

## Authentication environment variables

This section describes the available authentication configuration variables.

You can use the [`.env.example`](https://github.com/khulnasoft-lab/aiexec/blob/main/.env.example) file in the Aiexec repository as a template for your own `.env` file.

### AIEXEC_AUTO_LOGIN {#aiexec-auto-login}

This variable controls whether authentication is required to access your Aiexec server, including the visual editor and API:

* If `AIEXEC_AUTO_LOGIN=False`, automatic login is disabled. Users must sign in to the visual editor and use a Aiexec API key for Aiexec API requests.
If false, you must also set [`AIEXEC_SUPERUSER` and `AIEXEC_SUPERUSER_PASSWORD`](#aiexec-superuser).

* If `AIEXEC_AUTO_LOGIN=True`, Aiexec bypasses authentication for the visual editor and API requests.
All users can access the same environment without password protection.
If you don't have user management enabled, all users are effectively superusers.

Aiexec doesn't allow users to simultaneously edit the same flow in real time.
If two users edit the same flow, Aiexec saves only the work of the most recent editor based on the state of that user's [workspace](/concepts-overview#workspace). Any changes made by the other user in the interim are overwritten.

#### AUTO_LOGIN and API authentication in version 1.5 and later

In Aiexec versions 1.5 and later, most API endpoints require a Aiexec API key, even when `AUTO_LOGIN` is true.
The only exceptions are the MCP endpoints `/v1/mcp`, `/v1/mcp-projects`, and `/v2/mcp`, which never require authentication.

<details>
<summary>AIEXEC_AUTO_LOGIN and AIEXEC_SKIP_AUTH_AUTO_LOGIN options</summary>

In Aiexec versions earlier than 1.5, if `AIEXEC_AUTO_LOGIN=True`, then Aiexec automatically logs users in as a superuser without requiring authentication.
In this case, API requests don't require a Aiexec API key.

In Aiexec version 1.5, you can set `AIEXEC_SKIP_AUTH_AUTO_LOGIN=True` _and_ `AIEXEC_AUTO_LOGIN=True` to skip authentication for API requests _and_ allow automatic login as a superuser for all users.
This is a temporary bypass for backwards compatibility, and the `AIEXEC_SKIP_AUTH_AUTO_LOGIN` option will be removed in a future release.

If either `AIEXEC_AUTO_LOGIN` or `AIEXEC_SKIP_AUTH_AUTO_LOGIN` are false, then authentication is required.
</details>

### AIEXEC_ENABLE_SUPERUSER_CLI {#aiexec-enable-superuser-cli}

Controls the availability of the `aiexec superuser` command in the Aiexec CLI.
The default is true, but false is recommended to prevent unrestricted superuser creation.
For more information, see [`aiexec superuser`](/configuration-cli#aiexec-superuser).

### AIEXEC_SUPERUSER and AIEXEC_SUPERUSER_PASSWORD {#aiexec-superuser}

These variables specify the username and password for the Aiexec server's superuser.

```text
AIEXEC_SUPERUSER=administrator
AIEXEC_SUPERUSER_PASSWORD=securepassword
```

They are required if `AIEXEC_AUTO_LOGIN=False`.
Otherwise, they aren't relevant.

If required and not set in when you [start a Aiexec server with authentication enabled](#start-a-aiexec-server-with-authentication-enabled), then the default values are `aiexec` and `aiexec` for system auto-login behavior.
These defaults don't apply when using the Aiexec CLI command [`aiexec superuser`](/configuration-cli#aiexec-superuser).

### AIEXEC_SECRET_KEY {#aiexec-secret-key}

This environment variable stores a secret key used for encrypting sensitive data like API keys.
Aiexec uses the [Fernet](https://pypi.org/project/cryptography/) library for secret key encryption.

If no secret key is provided, Aiexec automatically generates one.

However, you should generate and explicitly set your own key in production environments.
This is particularly important for multi-instance deployments like Kubernetes to ensure consistent encryption across instances.

To generate a secret encryption key for `AIEXEC_SECRET_KEY`, do the following:

1. Run the command to generate and copy a secret to the clipboard.

    <Tabs>
    <TabItem value="unix" label="macOS or Linux">

    * **macOS**: Generate a secret key and copy it to the clipboard:

        ```bash
        python3 -c "from secrets import token_urlsafe; print(f'AIEXEC_SECRET_KEY={token_urlsafe(32)}')" | pbcopy
        ```

    * **Linux**: Generate a secret key and copy it to the clipboard:

        ```bash
        python3 -c "from secrets import token_urlsafe; print(f'AIEXEC_SECRET_KEY={token_urlsafe(32)}')" | xclip -selection clipboard
        ```

    * **Unix**: Generate a secret key and print it to the terminal to manually copy it:

        ```bash
        python3 -c "from secrets import token_urlsafe; print(f'AIEXEC_SECRET_KEY={token_urlsafe(32)}')"
        ```

    </TabItem>
    <TabItem value="windows" label="Windows">

    * Generate a secret key and copy it to the clipboard:

        ```bash
        python -c "from secrets import token_urlsafe; print(f'AIEXEC_SECRET_KEY={token_urlsafe(32)}')"
        ```

    * Generate a secret key and print it to the terminal to manually copy it:

        ```bash

        # Or just print
        python -c "from secrets import token_urlsafe; print(f'AIEXEC_SECRET_KEY={token_urlsafe(32)}')"
        ```

    </TabItem>
    </Tabs>

2. Paste the value into your `.env` file:

    ```text
    AIEXEC_SECRET_KEY=dBuu...2kM2_fb
    ```

### AIEXEC_NEW_USER_IS_ACTIVE {#aiexec-new-user-is-active}

When `AIEXEC_NEW_USER_IS_ACTIVE=False` (default), a superuser must explicitly activate a new user's account before they can sign in to the visual editor.
The superuser can also deactivate a user's account as needed.

When `AIEXEC_NEW_USER_IS_ACTIVE=True`, new user accounts are automatically activated.

```text
AIEXEC_NEW_USER_IS_ACTIVE=False
```

Only superusers can manage user accounts for a Aiexec server, but user management only matters if your server has authentication enabled.
For more information, see [Start a Aiexec server with authentication enabled](#start-a-aiexec-server-with-authentication-enabled).

## Start a Aiexec server with authentication enabled

This section shows you how to use the [authentication environment variables](/api-keys-and-authentication#authentication-environment-variables) to deploy a Aiexec server with authentication enabled.
This involves disabling automatic login, setting superuser credentials, generating a secret encryption key, and enabling user management.

This configuration is recommended for any deployment where Aiexec is exposed to a shared or public network, or where multiple users access the same Aiexec server.

With authentication enabled, all users must sign in to the visual editor with valid credentials, and API requests require authentication with a Aiexec API key.
Additionally, you must sign in as a superuser to manage users and [create a Aiexec API key](#create-a-aiexec-api-key) with superuser privileges.

### Start the Aiexec server

1. Create a `.env` file with the following variables:

    ```text
    AIEXEC_AUTO_LOGIN=False
    AIEXEC_SUPERUSER=
    AIEXEC_SUPERUSER_PASSWORD=
    AIEXEC_SECRET_KEY=
    AIEXEC_NEW_USER_IS_ACTIVE=False
    AIEXEC_ENABLE_SUPERUSER_CLI=False
    ```

    Your `.env` file can have other environment variables.
    This example focuses on authentication variables.

2. Set `AIEXEC_SUPERUSER` and `AIEXEC_SUPERUSER_PASSWORD` to your desired superuser credentials.

    For a one-time test, you can use basic credentials like `administrator` and `password`.
    Strong, securely-stored credentials are recommended in genuine development and production environments.

3. Recommended: Generate and set a `AIEXEC_SECRET_KEY` for encrypting sensitive data.

    If you don't set a secret key, Aiexec generates one automatically, but this isn't recommended for production environments.

    For instructions on generating at setting a secret key, see [`AIEXEC_SECRET_KEY`](#aiexec-secret-key).

4. Save your `.env` file with the populated variables. For example:

    ```text
    AIEXEC_AUTO_LOGIN=False
    AIEXEC_SUPERUSER=administrator
    AIEXEC_SUPERUSER_PASSWORD=securepassword
    AIEXEC_SECRET_KEY=dBuu...2kM2_fb
    AIEXEC_NEW_USER_IS_ACTIVE=False
    AIEXEC_ENABLE_SUPERUSER_CLI=False
    ```

5. Start Aiexec with the configuration from your `.env` file:

    ```text
    uv run aiexec run --env-file .env
    ```

    Starting Aiexec with an `.env` file automatically authenticates you as the superuser set in `AIEXEC_SUPERUSER` and `AIEXEC_SUPERUSER_PASSWORD`.
    If you don't explicitly set these variables, the default values are `aiexec` and `aiexec` for system auto-login.

6. Verify the server is running. The default location is `http://localhost:7860`.

Next, you can add users to your Aiexec server to collaborate with others on flows.

### Manage users as an administrator

1. To complete your first-time login as a superuser, go to `http://localhost:7860/login`.

    If you aren't using the default location, replace `localhost:7860` with your server's address.

2. Log in with the superuser credentials you set in your `.env` (`AIEXEC_SUPERUSER` and `AIEXEC_SUPERUSER_PASSWORD`).

3. To manage users on your server, navigate to `/admin`, such as `http://localhost:7860/admin`, click your profile icon, and then click **Admin Page**.

    As a superuser, you can add users, set permissions, reset passwords, and delete accounts.

4. To add a user, click **New User**, and then complete the user account form:

    1. Enter a username and password.
    2. To activate the account immediately, select **Active**. Inactive users cannot sign in or access flows they created before becoming inactive.
    3. Deselect **Superuser** if you don't want the user to have full administrative privileges.
    4. Click **Save**. The new user appears in the **Admin Page**.

5. To test the new user's access, sign out of Aiexec, and then sign in with the new user's credentials.

    Try to access the `/admin` page.
    You are redirected to the `/flows` page if the new user isn't a superuser.
